## 项目是什么

**DesignPatterns** 是一个 .NET 设计模式工具库：

- **运行时**：轻量、可组合的 primitives（责任链、策略、工厂注册表等），组合优于继承。
- **编译期（规划）**：Roslyn 源生成器、诊断器、CodeFix，可选 Completion；减少样板并在编译期发现误用。

不要把它做成「23 种模式的厚重框架」或 MediatR/Polly 的替代品。

## 实现新功能时的规则

### 运行时 API

- 保持 **最小依赖**；不要把 `Microsoft.Extensions.DependencyInjection` 拉进 Core。
- 公共 API 使用 XML 注释；异步方法以 `Async` 结尾，接受 `CancellationToken`。
- 优先接口 + 扩展方法 + 委托，避免深层 `abstract` 基类。
- 解析/查找用 `Try*` 或明确异常，避免无声失败。

### Roslyn（`DesignPatterns.SourceGenerators` / 规划中的 Analyzers）

- 生成器：**`IIncrementalGenerator`** + `ForAttributeWithMetadataName`。
- 分析器项目属性：`IsRoslynComponent=true`，`EnforceExtendedAnalyzerRules=true`、`IncludeBuildOutput=false`（生成器项目已配置）。
- 诊断 ID 格式：`DP###`，在 `DiagnosticIds` 类中集中定义。
- 只生成**注册表、排序管道、强类型 key 常量**等胶水；不生成完整业务状态机或领域 Builder。
- IDE 体验：优先生成 `StrategyKeys` 等常量；CompletionProvider 维护成本高，非必要不新增。

#### 源生成器：优先 `SyntaxFactory`，避免字符串拼接

编写生成代码时，**尽可能使用 Roslyn 语法 API**（`SyntaxFactory`、`SyntaxList`、`CompilationUnitSyntax` 等）构建 `SyntaxNode` / `CompilationUnit`，再经 `NormalizeWhitespace()` 输出；**不要**用 C# 源码字符串拼接、插值或大块 `"""` 模板生成逻辑。

推荐做法：

- 用 `SyntaxFactory` 构造类型、成员、语句、表达式；类型名用 `ParseTypeName` / `IdentifierName` 等，保持与语义模型一致。
- 将 `CompilationUnit` / `MemberDeclaration` 交给 `CSharpSyntaxTree.Create(...).GetRoot()`，通过 `AddSource` 注册。
- 需要格式化时依赖 `NormalizeWhitespace()`，必要时再配置 `Formatting.Whitespace` 相关选项。

仅在以下情况可例外（且需在 PR 中说明理由）：

- 固定的文件头注释（`// <auto-generated />` 等一行元数据）。
- 极小的、无分支的常量片段，且已有对应语法树测试覆盖。

避免字符串拼接的原因：易因转义、缩进、语言版本语法导致**生成代码无法编译**；难以重构；无法利用 Roslyn 的结构化校验。现有 `StrategyKeysGenerator` 中的 marker 仅为占位，新功能应遵循本规则。

相关 API：`Microsoft.CodeAnalysis.CSharp.SyntaxFactory`、`Microsoft.CodeAnalysis.CSharp.CSharpSyntaxTree`。

### 文件与命名

- 命名空间根：`DesignPatterns`；模式增多后再用 `DesignPatterns.Behavioral` 等子命名空间。
- 新项目名称与文件夹与现有 `DesignPatterns` 项目风格一致（PascalCase）。

## 当前里程碑（优先顺序）

## 不要做

- 不要提交 `bin/`、`obj/`、`.vs/`（已在 `.gitignore`）。
- 不要未经用户要求执行 `git commit` / `git push`。
- 不要在 Core 中引入重量级第三方框架。
- 不要用反射在运行时扫描「整个 AppDomain」替代编译期注册（与生成器设计冲突）。
- 不要为教学目的添加大段注释替代 `samples/` 示例项目。
- 不要过度工程：避免单用途一行 helper、无意义的抽象层。
- 不要在源生成器中用字符串拼接生成 C# 源码（见上文 SyntaxFactory 规则）；`RegisterStrategy` 等应用语法树实现。

## 文档维护

- 架构或工作流变更时，同步更新 `docs/DEVELOPMENT.md` 与本文件的相关章节。
- 每个新模式落地时：在 `samples/` 增加最小示例（项目存在后），并在 DEVELOPMENT 优先级表中勾选。

## 测试要求

- 运行时逻辑：xUnit（或仓库既定框架）单元测试。
- 生成器/分析器：Roslyn 测试包 + 快照；改动生成代码必须更新快照。
- 提交前至少：`dotnet build DesignPatterns.slnx` 通过。

## 与用户沟通

- 用户规则偏好：**最小 diff**、匹配现有风格、不主动 commit。
- 解释设计权衡时用中文（若用户用中文提问）；代码标识符与提交信息可用英文。
